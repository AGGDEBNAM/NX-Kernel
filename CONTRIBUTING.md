# Contributing to NX Platform

Thank you for your interest in contributing to NX Platform.

Our goal is to build a maintainable, scalable, and well-documented platform. Every contribution should follow the engineering standards defined for this project.

---

# Engineering Principles

When contributing, prioritize:

- Simplicity over complexity
- Engineering before implementation
- Modular design
- Consistency
- Long-term maintainability
- Documentation alongside code

---

# Development Workflow

Every significant change should follow this workflow:

1. Design Review
2. Engineering Specification
3. Implementation
4. Engineering Review
5. Integration
6. Merge

---

# Branch Strategy

Use descriptive branch names.

Examples:

feature/kernel-bootstrap

feature/runtime-services

docs/readme

docs/changelog

fix/logger

refactor/context

---

# Commit Convention

NX Platform follows Conventional Commits.

Examples:

feat(kernel): add bootstrap initialization

fix(runtime): resolve service registration bug

docs(readme): update architecture overview

refactor(context): simplify configuration loading

ci(github): add workflow validation

---

# Pull Request Process

Before opening a Pull Request:

- Verify the project builds successfully.
- Update documentation if required.
- Keep commits focused and descriptive.
- Ensure no unnecessary files are included.
- Describe the purpose of the change.

---

# Code Quality

Contributors should:

- Write readable code.
- Prefer simple solutions.
- Avoid unnecessary dependencies.
- Keep modules independent whenever possible.
- Maintain consistency with the existing architecture.

---

# Documentation

Documentation is considered part of the implementation.

Any architectural or behavioral change should be reflected in the appropriate documentation.

---

# Reporting Issues

When reporting a bug or requesting a feature, provide:

- Description
- Expected behavior
- Current behavior
- Environment (if applicable)
- Additional context

---

Thank you for helping improve NX Platform.
