# Objetivo

Documentar el incidente técnico detectado durante el Sprint **K1.2**, su análisis, la causa raíz, la solución aplicada y las reglas de ingeniería derivadas para evitar su recurrencia.

---

# Resumen Ejecutivo

Durante la implementación del Composition Root se presentó un error de ejecución al registrar servicios mediante `Register-NXService`.

Inicialmente el problema parecía estar relacionado con:

- Exportación de módulos.
- Orden de importación.
- Scopes de PowerShell.

Después de una investigación sistemática se identificó que la causa real era el uso de una expresión compleja (`[PSCustomObject]@{}`) directamente como argumento de un parámetro.

---

# Síntoma

Durante la inicialización del Kernel se obtenía un error similar a:

```text
A positional parameter cannot be found that accepts argument...
```

El fallo ocurría al ejecutar:

```powershell
Register-NXService `
    -Name "Configuration" `
    -Instance [PSCustomObject]@{
        ...
    }
```

# Evidencia

Se construyó una prueba mínima (MRE) reproduciendo el comportamiento del motor de PowerShell.

## Caso 1

```powershell
Register-NXService `
    -Name "Test" `
    -Instance @{}
```

Resultado

PASS

---

## Caso 2

```powershell
Register-NXService `
    -Name "Test" `
    -Instance [PSCustomObject]@{}
```

Resultado

FAIL

---

## Caso 3

```powershell
Register-NXService `
    -Name "Test" `
    -Instance ([PSCustomObject]@{})
```

Resultado

PASS

---

## Caso 4

```powershell
$instance = [PSCustomObject]@{}

Register-NXService `
    -Name "Test" `
    -Instance $instance
```

Resultado

PASS

---

# Causa Raíz

PowerShell no siempre interpreta correctamente expresiones complejas utilizadas directamente como valor de un parámetro.

Aunque la sintaxis es válida, el motor de Parameter Binding puede interpretar parte de la expresión como argumentos posicionales.

No se trató de un error del módulo ni de `Register-NXService`, sino de una característica del parser y del proceso de enlace de parámetros.

---

# Solución Aplicada

Se eliminó el uso de expresiones complejas inline.

Antes:

```powershell
Register-NXService `
    -Name "Configuration" `
    -Instance ([PSCustomObject]@{
        ...
    })
```

Después:

```powershell
$configurationService = [PSCustomObject]@{
    ...
}

Register-NXService `
    -Name "Configuration" `
    -Instance $configurationService
```

---

# Justificación

Aunque envolver la expresión entre paréntesis resuelve el problema técnico, utilizar una variable explícita ofrece ventajas adicionales:

- Mayor legibilidad.
- Mejor capacidad de depuración.
- Facilidad para inspeccionar objetos.
- Código más mantenible.
- Menor complejidad visual.

Por esta razón, el estándar del proyecto favorece variables temporales sobre expresiones inline.

---

# Reglas de Ingeniería

A partir de esta nota se establecen las siguientes reglas.

## Regla 1

No utilizar expresiones complejas directamente como argumentos de parámetros.

Evitar:

```powershell
-Instance [PSCustomObject]@{}
```

---

## Regla 2

Crear primero el objeto.

Después pasarlo al parámetro.

---

## Regla 3

Priorizar variables descriptivas.

Ejemplo:

```powershell
$configurationService
$loggerService
$runtimeContext
```

Evitar:

```powershell
$temp
$obj
$x
```

---

## Regla 4

Si un parámetro recibe un objeto complejo, éste deberá construirse antes de la llamada.

---

## Regla 5

Las llamadas a funciones deben permanecer simples.

Preferir:

```powershell
Register-NXService `
    -Name "Logger" `
    -Instance $loggerService
```

sobre expresiones anidadas.

---

# Lecciones Aprendidas

Durante la investigación se identificaron varios principios útiles.

- No asumir que un Parser Error implica un problema de sintaxis.
- No modificar arquitectura sin evidencia.
- Separar claramente síntomas de causas.
- Validar cada hipótesis mediante pruebas reproducibles.
- Construir siempre un MRE antes de aplicar una corrección definitiva.

# Acciones Permanentes

Esta Engineering Note introduce las siguientes prácticas obligatorias:

- Variables intermedias para objetos complejos.
- Validaciones mediante pruebas mínimas reproducibles.
- Correcciones mínimas antes de refactorizar.
- Documentar la causa raíz antes de cerrar un incidente.
