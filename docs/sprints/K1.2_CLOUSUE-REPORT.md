# Resumen Ejecutivo

El Sprint K1.2 tuvo como objetivo consolidar la arquitectura base del Kernel de NX Platform, estabilizando el proceso de inicialización, definiendo correctamente el Composition Root e implementando un modelo consistente de Dependency Injection.

Durante el Sprint se detectaron dos incidencias relevantes que permitieron fortalecer significativamente la arquitectura del proyecto.

El resultado final es un Kernel funcional, un Composition Root consolidado y un conjunto de estándares de ingeniería documentados para futuras iteraciones.

# Incidentes Detectados

## Bug #001

### Nombre

Register-NXService Parameter Binding

### Estado

Resolved

### Causa Raíz

Uso de expresiones complejas (`[PSCustomObject]`) directamente como argumentos de parámetros.

### Solución

Uso de variables intermedias.

---

## Bug #002

### Nombre

New-NXContext Runtime Resolution

### Estado

Resolved

### Causa Raíz

El Kernel construía el Context en lugar de recibirlo mediante Dependency Injection.

### Solución

El Composition Root pasó a crear, registrar e inyectar el Context.

---

# Decisiones Arquitectónicas

Durante el Sprint se aprobó oficialmente:

## ADR-0002

Context Lifecycle Owned by Composition Root.

Principios adoptados:

- Composition Root.
- Dependency Injection.
- Single Responsibility Principle.
- Separation of Concerns.

---

# Cambios Implementados

## NX.psm1

Se añadieron las siguientes responsabilidades:

- Construcción del Runtime.
- Creación del Context.
- Registro del Context.
- Inyección del Context.
- Coordinación del Bootstrap.

---

## Kernel.psm1

Se eliminaron responsabilidades relacionadas con:

- Creación del Context.
- Resolución de dependencias.

Ahora únicamente consume dependencias previamente construidas.

---

## Bootstrap

Se mantuvo como responsable de:

- Registro de servicios.
- Inicialización de la plataforma.
- Administración del ciclo de vida.
